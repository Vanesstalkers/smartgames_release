# Корпоративная игра (gameType: corporate)

## 1) Модель данных и исполнения

- Corporate реализована как **super-game** (`domain/game/corporate/class.js`) с вложенными под-играми команд (`classGame`).
- Super-game хранит:
  - `gamesMap` - соответствие команда -> игроки;
  - `roundPool` - план активных команд по раундам (competition);
  - `turnOrder` + `roundActiveGameId`/`roundActivePlayerId` - порядок хода после слияния (cooperative).
- Под-игра хранит командное состояние (`playerMap`, `round`, `roundReady`), а super-game управляет межкомандной координацией.

## 2) Подключение игрока: архитектурные инварианты

- `playerJoin` переопределен для двух сценариев:
  - **новый игрок**: выбор/создание слота в целевой под-игре и синхронизация `gamesMap`/`playerMap`;
  - **restoredPlayer**: повторное подключение существующего `playerId` без повторного конструирования состояния.
- Критический инвариант: в corporate-ветке обязательно вызывается `user.joinGame(...)`.
  - Это единственная точка, где консистентно выставляются `user.gameId` и session-связки.
  - `game.api.enter` опирается на этот инвариант (`gameId === user.gameId`).

## 3) Раундовая оркестрация

- Раунд запускается super-game (`roundStart`), а внутренняя механика команды выполняется в под-играх (`domain.roundStart`/`roundSteps`/`roundEnd`).
- **competition**:
  - активные команды выбираются через `roundPool.next({ fixState: true })`;
  - завершение супер-раунда происходит только когда все команды из фиксированного набора выставили `roundReady`.
- **cooperative**:
  - до полного слияния команды проходят раунды автономно;
  - после слияния активная команда выбирается из `turnOrder` (`selectNextActiveGame`).
- Переход к следующему раунду всегда инициируется super-game после проверки агрегированной готовности.

## 4) Восстановление и откат

- Источник восстановления - `game_dump` (снимки состояния по раундам).
- `loadGame` для corporate использует доменный `processData`, который реконструирует:
  - super-game;
  - все под-игры команд;
  - `gamesMap`/игроков;
  - `roundPool` и его активный ключ.
- После загрузки включается `restorationMode`; продолжение игры переводится через `restart()` без повторного сценария создания игры.
- `restoreForced` для corporate откатывает игру по `round`: удаляет runtime-экземпляр, загружает снимок, затем повторно подключает готовых участников через `playerJoin`/`viewerJoin`.

## 5) Ключевые архитектурные ограничения

- Согласованность `gamesMap` (super-game) и `playerMap` (под-игры) обязательна.
- Логика завершения раунда в competition зависит от **фиксированного** состава `roundPool.current(...)`, а не от динамически изменившегося списка команд.
- `restorationMode` меняет путь старта: игра продолжается через `restart()`, а не через обычную инициализацию новой партии.
